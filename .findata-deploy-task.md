너는 지금 `argocd-manifests` repo(ArgoCD GitOps 매니페스트)에 있다. 새 Dagster code location **findata**를 클러스터에 배포하도록 매니페스트를 작성/수정하는 게 임무다. 아래 컨텍스트는 자매 repo `dagster-pipelines`(별도)에서 넘어온 것이다.

## 배경
- `dagster-pipelines`에 새 프로젝트 `projects/findata`(금융 데이터 수집 파이프라인)가 추가됐고, 곧 main 머지되면 CI가 이미지 `cookieshake/findata-dagster`를 빌드/push한다.
- code-server 진입 모듈: `findata.definitions`, gRPC 포트 `4000`.
- 런타임에 필요한 env: `FINDATA_PG_DSN`(findata 데이터용 Postgres), `KIS_APP_KEY`/`KIS_APP_SECRET`/`KIS_CANO`/`KIS_HTS_ID`/`KIS_ENV`(한국투자증권 API 자격증명). asset 실행(run pod)에서 KIS·Postgres를 호출한다(code-server 자체는 정의만 로드).

## 이 repo의 관련 파일 (먼저 읽어라)
- `clusters/home/argo-applications/dagster.yaml` — dagster helm Application. code location은 `spec.source.helm.valuesObject.dagster-user-deployments.deployments[]`에 정의된다. 현재 옛 `news` 항목 하나만 있다(리네임 잔재 — 참고만, 이번 임무 범위는 findata 추가).
- `manifests/dagster/postgresql-secret.yaml`, `manifests/dagster/s3-secret.yaml` — Secret 작성 패턴(namespace `dagster`, `stringData`, placeholder 값 + 클러스터에 실제값 수동 주입).
- 같은 파일의 `dagster-configs` Application `ignoreDifferences` — placeholder Secret이 sync 때 실제값을 덮지 않게 하는 패턴.

## 해야 할 일
1. **findata code location 추가**: `dagster-user-deployments.deployments[]`에 `news`와 같은 형식으로 항목 추가.
   - `name: findata`
   - `image.repository: zot.ingtra.net/library/findata-dagster`, `tag: "bb539d0"`, `pullPolicy: Always`
     (사내 zot 레지스트리에 **멀티아치(amd64+arm64)**로 build/push 완료. Docker Hub/GitHub CI 안 씀. 어느 노드 arch든 OK.)
   - `dagsterApiGrpcArgs: ["-m", "findata.definitions"]`
   - `port: 4000`
   - 위 env 6개를 주입. 자격증명은 아래 새 Secret을 `envSecrets`로 참조.
   - **주의**: dagster helm 차트의 `dagster-user-deployments.deployments[]` 스키마에 맞는 정확한 필드명을 쓸 것(`envSecrets`, `env`, 필요 시 `volumes`/`volumeMounts`). 차트 버전은 Application의 `targetRevision: 1.13.10`. 스키마가 불확실하면 추측하지 말고 dagster helm 차트 values 스키마를 확인하거나 나(사용자)에게 물어라.
2. **Secret 매니페스트 신규**: `manifests/dagster/findata-secret.yaml` — `postgresql-secret.yaml` 패턴 그대로. `name: dagster-findata-secret`, namespace `dagster`, `stringData`에 placeholder 값으로 6개 키(`FINDATA_PG_DSN`, `KIS_APP_KEY`, `KIS_APP_SECRET`, `KIS_CANO`, `KIS_HTS_ID`, `KIS_ENV`). 실제값은 클러스터에 수동/외부 주입되므로 placeholder만.
3. **ignoreDifferences 추가**: `dagster-configs` Application의 `ignoreDifferences`에 `dagster-findata-secret`을 기존 두 secret과 같은 형식으로 추가(placeholder가 실제값을 덮어쓰지 않도록).

## 반드시 사용자에게 surface 할 설계 이슈 (혼자 임의로 결정하지 말 것)
- **KIS 토큰 캐시 vs 동시성**: findata는 유니버스를 ~128 해시 버킷으로 나눠 **버킷마다 별도 run pod**로 돌린다(K8sRunLauncher). tooja는 KIS 토큰을 디스크 캐시(platformdirs `~/.cache/tooja`)하는데, **KIS 토큰 발급은 분당 1회 레이트리밋**이 관측됐다. 128개 run pod가 각자 재인증하면 토큰 한도를 초과한다. 순수 PVC(RWO)는 다수 동시 pod에 못 붙는다(RWX 필요). → 이건 실제 인프라 설계 판단이 필요하니, 매니페스트에 임시안(예: 동시성 제한, RWX 볼륨, 또는 토큰 공유 방식)을 넣기 전에 **사용자에게 옵션을 제시하고 결정을 받아라.**

## 작업 방식·제약
- 변경은 만들되, **git commit/push는 사용자 확인 후에만.** 별도 인프라 repo이므로 함부로 push하지 마라.
- 먼저 관련 파일을 읽고, 변경 계획을 짧게 제시한 뒤 진행해라.
- 끝나면 무엇을 바꿨는지 요약하고, 위 토큰-캐시 이슈 결정과 다음 배포 단계(PR 머지→이미지 빌드→DB 생성/`apply_schema()`→실제 secret 값 주입)를 정리해줘.

작업 완료하거나 사용자 결정이 필요하면, 이 repo의 사용자에게 보고하면 된다.
